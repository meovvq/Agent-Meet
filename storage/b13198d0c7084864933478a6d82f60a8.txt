# AI 工程实践常见问题详解

---

## 一、Token 基础与上下文窗口

### 1. Token 是什么？为什么中文、英文、代码消耗的 Token 不一样？

**Token** 是大语言模型处理文本的基本单位，可以理解为模型"阅读"和"书写"的最小语义片段。它可以是一个完整的单词（如 `hello`）、一个汉字（如 `你`）、一个标点符号（如 `，`），也可以是单词的一部分（如 `un-`、`-ness`）(citation:6)。

模型在接收文本时，会先将其拆分为一串 token 序列，再转化为数值 ID 进行计算，最后将预测出的 token 再转换回文本(citation:6)。

**不同语言的 token 消耗差异，根本原因在于分词算法的编码方式不同：**

- **英文**：采用字节级 BPE（Byte-level BPE）等子词分词算法，高频词通常保持完整，经验法则是 1 个 token ≈ 4 个字符 ≈ ¾ 个单词(citation:6)。
- **中文**：由于中文没有空格分隔，原始 BPE 需按 UTF-8 字节拆分（1 个汉字 = 3 字节），导致 token 数增加。经验数据显示，中文 1 字约对应 1.5～2.5 个 token(citation:7)。不过部分国内模型如通义千问、文心一言做了深度优化，可实现约 1 汉字 = 1 token 的映射(citation:7)。
- **代码**：代码中包含大量标点符号（`{`、`}`、`;`）、缩进空格和重复关键字，这些都会被单独切分为 token。加上变量名可能是低频子词组合，因此代码的 token 消耗通常高于同等长度的自然语言(citation:7)。

不同模型的分词器实现不同，因此同一段文本在不同模型中的 token 数也会有差异(citation:8)。

---

### 2. 上下文窗口是什么？上下文窗口越大，效果一定越好吗？

**上下文窗口（Context Window）** 指模型在单次推理中能"看到"的最大 token 总量，包括输入（系统提示、用户消息、历史对话、RAG 证据等）和输出(citation:6)。

当前主流模型的上下文窗口差异显著。例如 Gemma 4 的小型模型支持 12.8 万 token，中型模型支持 25.6 万 token(citation:1)；Gemini 系列支持到约 100 万 token(citation:9)。

**上下文窗口越大，效果不一定越好，原因如下：**

1. **Lost in the Middle 问题**：模型对长上下文中间部分的信息关注度会显著下降（详见下一题）(citation:3)。
2. **成本与延迟**：更长的上下文意味着更多的输入 token，推理成本和延迟都会增加(citation:6)。
3. **注意力稀释**：当上下文过长时，模型可能无法有效区分关键信息和噪声，导致回答质量下降。
4. **实际需求有限**：很多场景下几千 token 已足够，盲目使用长上下文是浪费资源。

最佳实践是：根据实际需求选择合适的上下文长度，并通过合理的提示设计和信息筛选来提高信息密度。

---

### 3. 什么是 Lost in the Middle 问题？长上下文场景下怎么缓解？

**Lost in the Middle（中间丢失）** 是指模型在处理长文本时，对放置在上下文开头和结尾的信息关注度较高，而对中间部分的信息利用率显著下降的现象(citation:3)。这在长输入和长输出任务中尤为突出。

**缓解方案包括：**

1. **检索增强 + 位置优化**：在构建 RAG 上下文时，将最相关的信息放在提示的开头或结尾，而不是中间(citation:3)。
2. **RAL-WRITER 方法**：论文提出了一种检索增强长文本生成器，包含"写作步骤规划器"和"检索及重述编写器"两个模块。通过基于块的检索机制，评估每个文本块的重要性，并引入位置得分来减少中间丢失(citation:3)。
3. **分段处理**：将长文本拆分为多个段落，分别处理后再汇总，避免一次性输入过长内容。
4. **信息重排**：对检索到的文档进行 Re-ranking，把最相关的内容排在前面，提升模型关注度(citation:5)。
5. **使用支持更大上下文的模型**：如 GPT-5.2 Thinking 在 4-needle MRCR 评测（256k token）中实现了接近 100% 的准确率(citation:2)，说明新模型在长上下文处理上已有显著改进。

## 二、采样参数与输出控制

### 4. Temperature、Top-P、Top-K 分别控制什么？生产环境怎么设置更稳？

这三个参数作用于模型采样流水线的不同环节(citation:10)：

| 参数 | 作用 | 直觉理解 |
|------|------|----------|
| **Temperature** | 在 softmax 之前对 logits 进行缩放，控制概率分布的尖锐程度 | 温度低 → 分布尖锐 → 输出确定；温度高 → 分布平坦 → 输出多样 |
| **Top-P（核采样）** | 按概率排序后，只保留累积概率达到阈值 p 的最小 token 集合 | 自适应截断，模型自信时核很小，不确定时核扩大 |
| **Top-K** | 始终保留概率最高的 k 个 token | 固定截断，不考虑概率分布的实际形状 |

**生产环境建议设置**(citation:10)：

| 场景 | Temperature | Top-P | 说明 |
|------|------------|-------|------|
| 结构化抽取/JSON 生成 | 0.0-0.2 | 0.9 | 追求一致性，配合约束解码效果最佳 |
| 代码生成 | 0.2-0.5 | 0.9 | 需要语法精确，但允许一定探索 |
| 多步推理 | 0.5-0.7 | 0.9 | temperature=0 反而可能降低复杂推理准确率 |
| 摘要与问答 | 0.5-0.7 | 0.9 | 事实准确性与可读性的平衡点 |
| 创意写作 | 0.8-1.2 | 0.95 | 需要更多分布探索 |

Gemma 4 推荐的标准化配置为 `temperature=1.0`、`top_p=0.95`、`top_k=64`(citation:1)。

**关键原则**：将 temperature 和 top-p 视为耦合参数。提高 temperature 时应降低 top-p，否则分布过于平坦会导致语无伦次的输出(citation:10)。

---

### 5. 为什么 Temperature 设置为 0，模型输出仍然可能不完全一致？

Temperature=0 意味着贪心解码（始终选择概率最高的 token），但并不能保证确定性输出(citation:10)：

1. **浮点运算非确定性**：GPU 上的浮点运算不满足结合律，`(a+b)+c` 与 `a+(b+c)` 可能产生不同结果。在 transformer 的数百次矩阵乘法中，舍入误差会累积，可能翻转哪个 token 概率最高(citation:10)。
2. **连续批处理影响**：推理服务器将同时到达的请求批量处理，批次组成会影响注意力模式和 logits 输出。单独运行与和其他请求批量运行会产生不同结果(citation:10)。
3. **硬件和环境差异**：GPU 内存布局、量化变体、KV 缓存优化等都可能引入额外漂移。

如果需要真正的确定性输出（如审计日志、回归测试），需要固定批次大小、一致的硬件配置，以及平台级别的种子支持(citation:10)。

### 6. 大模型为什么会产生幻觉？常见缓解方案有哪些？

**幻觉（Hallucination）** 是指模型生成看似合理但实际上不正确或无依据的内容。

**产生原因**：
- 模型本质是基于概率的下一个 token 预测器，而非知识检索系统
- 训练数据中的噪声和偏差
- 模型倾向于生成流畅且自信的文本，即使信息不足也会"填补空白"
- 长上下文中间部分信息丢失加剧幻觉(citation:3)

**常见缓解方案**：

1. **RAG（检索增强生成）**：将模型回答"锚定"在可验证的、检索到的证据上，提高事实准确性(citation:5)。
2. **降低 Temperature**：较低的 temperature 减少分布探索，降低编造内容的概率(citation:10)。
3. **系统提示约束**：通过 system prompt 要求模型仅基于提供的上下文回答，不确定时声明"不知道"。
4. **多步验证**：让模型先生成答案，再用另一个调用验证答案是否有依据。
5. **选择更强的模型**：GPT-5.2 Thinking 的幻觉率比 GPT-5.1 Thinking 相对减少了 38%(citation:2)。
6. **Function Calling**：让模型调用工具获取实时数据，而非凭记忆回答(citation:1)。

## 三、Token 预算与上下文管理

### 7. Token 预算怎么估算？输入、输出、历史消息、RAG 证据如何取舍？

**Token 预算 = 上下文窗口大小 - 输出预留空间**

估算方法(citation:6)(citation:7)(citation:8)：
- 英文：1 token ≈ 4 字符 ≈ 0.75 个单词
- 中文：1 token ≈ 1～1.5 个汉字（因模型而异）
- 可使用 tiktoken 等工具精确计算(citation:6)

**取舍优先级**：

| 优先级 | 组件 | 策略 |
|--------|------|------|
| 1 | 系统提示 | 必须保留，但应精简，去除冗余指令 |
| 2 | 当前用户输入 | 必须完整保留 |
| 3 | RAG 证据 | 只检索最相关的片段，而非整个文档(citation:5) |
| 4 | 历史消息 | 采用滑动窗口或摘要压缩，保留最近 N 轮 |
| 5 | 输出空间 | 预留足够的输出 token，一般为输入的 20%-50% |

Gemma 4 支持可变分辨率的视觉 token 预算（70、140、280、560、1120），可根据任务复杂度选择不同精度(citation:1)。

---

### 8. 长上下文窗口会不会取代 RAG？二者分别适合什么场景？

**不会完全取代，二者是互补关系**。

| 维度 | 长上下文窗口 | RAG |
|------|------------|-----|
| **数据来源** | 将所有信息一次性塞入提示 | 按需检索，动态注入相关片段(citation:5) |
| **适用场景** | 需要全局理解的任务（如整本书分析、长文档对比） | 知识库频繁更新、数据量远超上下文窗口的场景 |
| **成本** | 输入 token 随长度线性增长 | 检索成本较低，但需要额外的索引和检索系统 |
| **信息质量** | 需自行筛选关键信息，存在 Lost in the Middle(citation:3) | 通过检索算法筛选最相关内容，信息密度更高 |
| **实时性** | 信息冻结在提示构建时刻 | 可检索最新数据，解决"知识冻结"问题(citation:5) |

**最佳实践**：多数生产系统采用 RAG 作为主要策略，长上下文窗口作为补充。当需要处理单个长文档的深度分析时使用长上下文；当需要从大规模知识库中按需获取信息时使用 RAG(citation:5)。

## 四、API 调用链路与流式输出

### 9. 大模型 API 调用的完整链路是什么？

一个典型的大模型 API 调用链路如下：

```
客户端 → API 网关 → 模型路由 → 上下文构建（RAG/历史拼接）
    → 采样参数配置 → 模型推理 → 输出过滤/安全检查
    → Token 计量与计费 → 响应返回客户端
```

具体步骤：
1. **客户端发起请求**：携带用户消息、系统提示、采样参数等
2. **API 网关处理**：鉴权、限流、日志记录、负载均衡
3. **上下文构建**：拼接系统提示、历史消息、RAG 检索结果
4. **模型推理**：将 token 序列输入模型，逐 token 生成输出
5. **后处理**：输出格式校验、安全过滤、敏感内容检测
6. **计量返回**：统计输入/输出 token 数，返回结果和用量元数据(citation:6)

---

### 10. Streaming 为什么能改善用户体验？它能减少总耗时和 Token 成本吗？

**Streaming 改善用户体验的原理**：模型是逐 token 生成的，首 token 生成后即可开始返回，用户不必等待全部生成完毕(citation:6)。

- **首字延迟（TTFT）**：从请求发出到收到第一个 token 的时间，streaming 让用户几乎立即看到输出开始。
- **感知速度**：用户看到文字逐渐出现，心理等待时间大幅缩短。

**能否减少总耗时和 Token 成本？**

- **总耗时**：不能减少模型生成全部 token 的实际时间，但通过"边生成边传输"消除了等待感。
- **Token 成本**：不能减少 token 消耗量。streaming 改变的是传输方式，不影响生成的 token 总数和计费(citation:6)。
- **网络层面**：对于长输出，streaming 可以避免一次性传输大量数据导致的超时风险。

---

### 11. SSE、WebSocket、HTTP Chunked 在流式输出场景下怎么选？

| 技术 | 方向 | 适用场景 | 优缺点 |
|------|------|----------|--------|
| **SSE（Server-Sent Events）** | 服务端单向推送 | LLM 流式输出的标准选择 | 实现简单、浏览器原生支持、自动重连；仅支持单向通信 |
| **WebSocket** | 全双工双向通信 | 需要实时交互的场景（如聊天应用中打断/编辑） | 支持双向通信，但实现复杂，连接管理成本更高 |
| **HTTP Chunked** | 服务端分块传输 | API 层面的底层传输 | 底层传输机制，SSE 建立在此之上；纯 Chunked 需要客户端自行解析边界 |

**推荐选择**：
- **大多数 LLM 流式输出场景** → 使用 **SSE**，它是 OpenAI、Anthropic 等主流 API 的默认流式协议。
- **需要双向实时交互** → 使用 **WebSocket**。
- **内部服务间通信** → 可使用 HTTP Chunked 以降低协议开销。

## 五、API 错误处理与可靠性

### 12. 品些大模型 API 错误可以重试？哪些错误不能重试？

| 错误类型 | HTTP 状态码 | 可否重试 | 策略 |
|----------|-----------|---------|------|
| 服务端内部错误 | 500 | 可重试 | 指数退避重试 |
| 服务暂时不可用 | 502/503 | 可重试 | 指数退避重试 |
| 限流/超配额 | 429 | 可重试 | 等待 Retry-After 后重试 |
| 请求超时 | 超时 | 可重试 | 重试前确认是否为幂等操作 |
| 认证失败 | 401 | **不可重试** | 检查 API Key |
| 参数错误 | 400 | **不可重试** | 修正请求参数 |
| 权限不足 | 403 | **不可重试** | 检查账户权限 |
| 模型不存在 | 404 | **不可重试** | 更换模型标识 |

**原则**：客户端错误（4xx）通常不应重试（除非是 429），服务端错误（5xx）和网络错误可以重试。重试时应采用指数退避策略，避免加剧服务端压力。

---

### 13. 为什么大模型调用必须做幂等？

大模型 API 调用天然具有**非确定性**（即使 temperature=0 也不完全一致）(citation:10)，因此"幂等"的含义与传统 API 略有不同，但仍然必须做，原因是：

1. **重试安全性**：网络抖动、超时等情况需要重试。如果调用不是幂等的，重试可能导致重复扣费、重复写入等后果。
2. **一致性保障**：在支付、订单等关联场景中，同一请求多次执行必须产生相同副作用。
3. **分布式系统可靠性**：消息队列、任务调度等场景下，消息可能被重复投递。

**实现方式**：
- 客户端生成唯一请求 ID（`idempotency_key`）
- 服务端基于该 ID 做去重，已处理的请求直接返回缓存结果
- 对于大模型特有的"非确定性"，可缓存第一次的完整响应结果

---

### 14. 大模型限流为什么不能只按 QPS 做？

传统 API 限流通常按 QPS（每秒请求数）计算，但大模型 API 有其特殊性：

1. **请求成本差异巨大**：一个发送 10 token 的简单查询与一个发送 10 万 token 的长文档分析，消耗的计算资源完全不同(citation:6)。按 QPS 限流无法反映实际资源消耗。
2. **输出长度不可预测**：模型可能生成 10 token 也可能生成 4000 token，对 GPU 的占用时间差异巨大。
3. **批处理影响**：推理服务器采用连续批处理(citation:10)，一个长上下文请求会"卡住"同批次的其他请求。

**更合理的限流维度**：
- **Token 速率**（tokens/minute）：反映实际计算量
- **并发请求数**（concurrent requests）：反映 GPU 并行能力
- **混合限流**：QPS + Token 速率 + 并发数的组合

---

### 15. 模型网关通常要承担哪些能力？

模型网关（Model Gateway）是大模型应用架构中的关键中间层，通常承担以下能力：

1. **统一鉴权**：对下游屏蔽多个模型提供商的认证方式，统一 API Key 管理
2. **限流与配额管理**：按用户/应用/模型维度做精细化限流（不只按 QPS）(citation:10)
3. **模型路由**：根据请求特征（任务类型、成本预算、延迟要求）路由到不同模型
4. **负载均衡**：在多个模型实例间分配请求
5. **重试与容错**：自动重试可恢复错误，实现熔断降级
6. **可观测性**：记录请求日志、监控延迟、统计 token 消耗
7. **安全治理**：输入输出内容过滤、敏感信息脱敏
8. **协议适配**：统一对接不同厂商的 API 差异（参数名、流式协议等）
9. **缓存**：对相同输入缓存响应结果，降低延迟和成本
10. **成本归集**：按团队/项目统计 token 消耗和费用

---

### 16. AI 应用的调用日志里至少要记录哪些字段？

| 字段类别 | 具体字段 | 说明 |
|----------|---------|------|
| **请求标识** | request_id, idempotency_key | 唯一标识，支持链路追踪和去重 |
| **时间信息** | request_time, first_token_time, response_time | 用于计算 TTFT 和总延迟 |
| **模型信息** | model_name, model_version | 记录使用的具体模型(citation:9) |
| **输入内容** | system_prompt, user_message, context_messages | 支持问题复现和质量分析 |
| **输出内容** | response_text, finish_reason | 记录模型实际输出和结束原因 |
| **Token 用量** | input_tokens, output_tokens, cached_tokens | 核心计量字段(citation:6) |
| **采样参数** | temperature, top_p, top_k | 记录实际使用的参数(citation:10) |
| **状态信息** | status_code, error_message | 请求是否成功 |
| **安全信息** | content_filter_results, sensitive_flags | 内容安全审查结果 |
| **业务上下文** | user_id, session_id, trace_id | 关联业务流程 |

## 六、结构化输出与工具调用

### 17. 为什么只写"请返回 JSON"不可靠？

1. **模型不保证格式**：自然语言指令存在模糊性，模型可能返回包含解释文字的 JSON、格式不正确的 JSON，或完全非 JSON 的内容。
2. **LLM 本质是概率生成**：每次生成都是逐 token 采样，即使绝大多数时候返回了有效 JSON，也有概率出现格式错误(citation:10)。
3. **特殊字符处理**：JSON 中的引号、换行符、转义字符等容易被模型"自由发挥"。
4. **复杂嵌套困难**：深层嵌套的 JSON 结构，模型在生成过程中更容易丢失括号配对。

**可靠方案**：使用 JSON Mode 或 Structured Outputs，从模型层面强制输出符合格式要求。

---

### 18. JSON Mode 和 Structured Outputs 有什么区别？

| 维度 | JSON Mode | Structured Outputs |
|------|-----------|-------------------|
| **保证程度** | 保证输出是有效 JSON，但不保证结构 | 保证输出严格符合预定义的 JSON Schema |
| **实现方式** | 启用后模型仅输出合法 JSON 文本 | 通过约束解码，在采样前从数学上排除不符合 schema 的 token(citation:10) |
| **灵活性** | 模型可以自由决定 JSON 的键和结构 | 必须完全匹配预定义的 schema，包括字段名、类型、必填性 |
| **适用场景** | 需要 JSON 格式但不限定具体结构 | 需要精确的、可解析的结构化数据 |
| **可靠性** | 较高，但仍可能不符合预期结构 | 极高，几乎 100% 符合 schema(citation:10) |

---

### 19. JSON Schema 在大模型应用里解决什么问题？

JSON Schema 在大模型应用中扮演**接口契约**的角色：

1. **输出格式约束**：明确告诉模型需要返回哪些字段、每个字段的类型、是否必填，避免模型"自由发挥"
2. **下游解析可靠性**：下游系统可以直接按 schema 解析，无需做容错处理
3. **Structured Outputs 的基础**：作为约束解码的输入，从数学上保证输出的有效性(citation:10)
4. **文档化与协作**：Schema 本身就是 API 文档，前端和后端可以基于同一份 schema 协作开发
5. **测试与验证**：可以用 schema 自动验证模型输出，检测格式错误

---

### 20. Function Calling 的完整链路是什么？

```
1. 客户端定义工具 → 提供工具名称、描述、参数的 JSON Schema
2. 用户发送消息 → 携带工具定义和用户输入
3. 模型决策 → 模型判断是否需要调用工具，如需要则生成工具调用请求
   （包含工具名、参数 JSON）
4. 客户端执行 → 应用层拿到工具调用请求，执行实际的 API/函数调用
5. 结果返回模型 → 将工具执行结果作为新消息发回模型
6. 模型整合输出 → 基于工具返回结果生成最终回答
```

整个过程中，模型**不直接执行**工具，只负责"决定调用什么、传什么参数"。实际执行由应用层完成(citation:1)。

---

### 21. Function Calling 和 MCP 有什么区别？

| 维度 | Function Calling | MCP（Model Context Protocol） |
|------|-----------------|-------------------------------|
| **层级** | 模型层面的能力 | 协议层面的标准 |
| **作用** | 让模型能够"请求"调用外部函数 | 定义模型与工具/数据源之间的标准化通信协议 |
| **工具定义** | 在每次请求中内联定义 | 通过 MCP Server 注册，支持动态发现 |
| **互操作性** | 绑定特定模型 API（各厂商格式不同） | 跨模型、跨平台的开放标准 |
| **关系** | MCP 依赖 Function Calling 作为模型侧的触发机制 | MCP 是 Function Calling 的"标准化封装层" |

简而言之，Function Calling 是"模型怎么请求调用工具"，MCP 是"工具怎么被标准化地暴露和管理"。

---

### 22. MCP Tool 和普通 HTTP API 有什么关系？

MCP Tool 本质上是对普通 HTTP API 的一层**标准化封装**：

- **底层实现**：MCP Tool 的实际功能通常就是一个 HTTP API 调用
- **协议转换**：MCP Server 将标准化的 MCP 协议请求转换为具体的 HTTP API 调用
- **差异**：
  - HTTP API 面向人类开发者设计，文档和接口格式各异
  - MCP Tool 面向 AI 模型设计，提供机器可理解的工具描述、参数 schema
- **价值**：MCP 让 AI 模型能够以统一的方式发现和使用各种工具，无需针对每个 API 编写专门的适配代码

---

### 23. Agent Skill 和 Function Calling 是一回事吗？

**不完全是一回事**：

- **Function Calling** 是底层的模型能力：模型输出结构化的工具调用请求(citation:1)
- **Agent Skill** 是上层的业务能力：一个 Skill 可能涉及多次 Function Calling + 中间推理 + 状态管理 + 错误处理

| 维度 | Function Calling | Agent Skill |
|------|-----------------|-------------|
| 粒度 | 单次工具调用 | 多步骤的任务完成流程 |
| 状态 | 无状态 | 可能维护上下文状态 |
| 错误处理 | 模型不处理 | 包含重试、降级、分支逻辑 |
| 例子 | "调用搜索 API 查询 X" | "处理一个完整的客服工单" |

一个 Agent Skill 可能包含多个 Function Calling 调用，加上规划、反思、记忆等组件。

---

### 24. 结构化输出失败后怎么处理？

1. **重试**：对于偶发性格式错误，直接重试（带上错误信息反馈给模型），通常 1-2 次重试可解决
2. **宽松解析**：尝试对模型输出做容错解析（如去除 markdown 代码块标记、修复常见 JSON 错误）
3. **降级处理**：重试失败后，降级到非结构化输出，由应用层自行提取关键信息
4. **反馈修正**：将解析错误信息作为新消息发回模型，请求修正（如"你的 JSON 缺少闭合括号，请修正"）
5. **切换方案**：如果频繁失败，考虑使用 Structured Outputs 或约束解码来从源头保证格式(citation:10)
6. **监控告警**：记录失败率，超过阈值时触发告警，排查是否为 prompt 设计问题

---

### 25. 工具调用为什么必须做安全治理？

1. **权限控制**：模型可能被诱导调用高权限工具（如删除数据、转账），必须有最小权限控制
2. **参数注入**：恶意用户可能通过提示注入，让模型构造危险的工具参数（如 SQL 注入式的查询参数）
3. **资源滥用**：模型可能在循环中反复调用工具，造成 API 调用量暴增和费用飙升
4. **数据泄露**：工具返回的结果可能包含敏感信息，需要在返回给用户前做过滤
5. **合规要求**：某些工具调用（如访问外部 API）需要审计日志和审批流程

**治理手段**：白名单机制、参数校验沙箱、调用频率限制、人工审批队列、敏感操作二次确认。

---

### 26. 面试里怎么一句话概括结构化输出？

> 结构化输出是通过约束解码或 Schema 定义，让大模型的输出从"自由文本"变为"可解析、可验证的结构化数据"，从而可靠地对接下游系统。

## 七、AI 应用评测

### 27. 为什么不能只靠公开 benchmark 评估 AI 应用质量？

1. **数据污染**：公开 benchmark 的测试数据可能已被纳入模型训练集，导致分数虚高(citation:2)
2. **场景不匹配**：benchmark 测试的是通用能力，而你的应用有特定的领域、数据和业务逻辑
3. **指标单一**：benchmark 通常只衡量准确率，但实际应用还需要考虑延迟、成本、稳定性、安全性等
4. **用户感知差异**：benchmark 分数高不代表用户体验好，人类评估与自动指标经常不一致(citation:2)
5. **Prompt 敏感性**：同一模型在不同 prompt 下表现差异巨大(citation:10)，benchmark 使用的 prompt 与你的不同

GPT-5.2 的发布就展示了这一点——它在多个 benchmark 上刷新纪录(citation:2)，但实际应用效果还需要在具体场景中验证。

### 28. Golden Set 应该怎么构建？冷启动阶段没有生产日志怎么办？

**Golden Set** 是一组带有标准答案的测试用例，用于评估模型在特定任务上的表现。

**构建方法**：

1. **领域专家标注**：邀请业务专家手工编写输入和期望输出
2. **生产日志采样**：从真实流量中抽取有代表性的案例（已有生产数据时）
3. **边界覆盖**：确保覆盖正常情况、边界情况、对抗性输入

**冷启动阶段的策略**：

1. **合成数据生成**：用 LLM 自动生成多样化的测试用例，再由人工审核和修正
2. **公开数据集迁移**：找领域相关的公开数据集，筛选和改编为你的场景
3. **少量高质标注**：先标注 50-100 个高质量样本覆盖核心场景，后续持续补充
4. **A/B 迭代**：用初始 golden set 跑出基线，上线后从生产日志中不断补充新案例
5. **多人标注 + 一致性检查**：同一案例多人独立标注，计算一致性（如 Cohen's Kappa），确保标注质量

### 29. LLM-as-Judge 有哪些主要偏差？怎么缓解？

用 LLM 评估 LLM 输出（LLM-as-Judge）是当前流行的评估方式，但存在以下偏差：

| 偏差类型 | 表现 | 缓解方法 |
|----------|------|---------|
| **位置偏差** | 倾向于选择列表中靠前或靠后的选项 | 随机化选项顺序，多次评估取平均 |
| **长度偏差** | 倾向于认为更长的回答质量更高 | 控制输出长度，或在评估提示中明确"长度不等于质量" |
| **自我偏好** | 倾向于偏好与自己风格相似的输出 | 使用不同模型交叉评估 |
| **风格偏差** | 偏好华丽/自信的表达，即使内容有误 | 强调"评估事实准确性而非文采" |
| **冗余偏差** | 对包含更多细节的回答评分更高，即使细节有误 | 明确评估维度和权重 |
| **格式偏差** | 对有良好格式（列表、标题）的输出评分更高 | 将格式和内容分开评估 |

**系统性缓解**：
- 使用多个不同的 Judge 模型
- 引入人工标注的 golden set 作为校准基准
- 采用 pairwise comparison（两两对比）而非绝对评分
- 定期用人工评估结果校准 LLM Judge

### 30. RAG 评测为什么必须分检索和生成两段？

RAG 系统的输出质量取决于两个独立环节的叠加效果(citation:5)：

```
用户查询 → [检索模块] → 相关文档片段 → [生成模块] → 最终回答
```

**必须分段评测的原因**：

1. **错误归因**：如果只看最终回答质量，无法判断错误来自检索（没找到正确文档）还是生成（找到了但没用好）。两种错误的修复方向完全不同。
2. **检索质量指标**：
   - 召回率（Recall）：是否找到了所有相关文档
   - 精确率（Precision）：检索结果中有多少是真正相关的
   - 排名质量（MRR/NDCG）：最相关的文档是否排在前面(citation:5)
3. **生成质量指标**：
   - 忠实度（Faithfulness）：回答是否忠于检索到的上下文
   - 相关性（Relevance）：回答是否切题
   - 完整性（Completeness）：是否涵盖了所有关键信息
4. **优化方向不同**：检索差需要优化分块策略、embedding 模型、检索算法；生成差需要优化 prompt 设计、模型选择、上下文组织(citation:5)。

**最佳实践**：先确保检索模块的召回率和精确率达到基线，再优化生成模块的 prompt 和参数。分段评测是精准优化的前提。
